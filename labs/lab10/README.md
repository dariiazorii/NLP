# Lab 10: Hybrid Named Entity Recognition (NER)

## 1. Корпус та Evaluation Set
* **Джерело:** Вибірка з 25 речень тендерних оголошень та юридичних документів українською мовою з корпусу 3015 документів Prozorro.
* **Склад:** Набір містить стандартні сутності (ПІБ, локації) та специфічні доменні дані (дати, грошові суми, коди закупівлі).
* **Метрики:** Оцінка проводилася на основі порівняння точних збігів (Exact Match) тексту та мітки.

## 2. Обраний Pipeline
Використано **uk_core_news_trf** (spaCy). Це Transformer-модель на базі RoBERTa, яка забезпечує глибоке розуміння контексту української мови та підтримує стандартні мітки OntoNotes (PERSON, ORG, GPE, LOC тощо).

## 3. Додані Hybrid Rules
Для виправлення "сліпих плям" моделі через компонент `EntityRuler` додано:
* **DATE:** Правила для числових (ДД.ММ.РРРР) та текстових форматів дат.
* **MONEY:** Логіка об'єднання числових токенів із символом валюти (грн).
* **ID/CPV:** Патерни для розпізнавання кодів класифікатора закупівлі та номерів наказів.
* **Dictionary ORG:** Фіксований список доменних абревіатур (НСЗУ, БВПД, КМУ).

## 4. Результати Baseline (що працювало добре)
* **PERSON:** Стабільно висока точність у розпізнаванні Прізвищ та Імен з великої літери.
* **GPE:** Чітке визначення великих міст та країн.
* **ORG:** Коректне маркування стандартних юридичних форм (ТОВ, ПАТ).

## 5. Слабкі сторони Baseline (що було пропущено)
* Повна відсутність розпізнавання дат у будь-якому форматі.
* Фрагментація грошових сум (число окремо, валюта окремо або ігнорування).
* Повне ігнорування технічних ідентифікаторів та кодів CPV.
* **Boundary Errors:** Відсікання номерів будинків в адресах (`LOC`).

## 6. Вплив правил (Hybrid Improvement)
* **Recall:** Показник повноти для дат та кодів зріс з 0% до ~90%.
* **Цілісність:** Сутності `MONEY` та `LOC` стали більш цілісними (менше розривів у межах сутностей).
* **Доменна адаптація:** Модель почала бачити специфічні абревіатури навіть у нижньому регістрі.

## 7. Невирішені проблеми
* **False Positives:** Агресивні правила іноді маркують звичайні цифри або слова як `ID`.
* **Ambiguity:** Складність у розрізненні, чи є число номером будинку, сумою чи кодом без додаткового контексту.
* **Токенізація:** Символи `№`, дефіси та крапки у великих числах все ще створюють Boundary Errors у складних випадках.
